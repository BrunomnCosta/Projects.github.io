01 O Problema The Problem

A maioria dos projetos de portfólio em cibersegurança demonstra conhecimento teórico — certificações, CTFs, laboratórios de cursos. O que falta é evidência de capacidade de construir infraestrutura de monitorização operacional de raiz.

Most cybersecurity portfolio projects demonstrate theoretical knowledge — certifications, CTFs, course labs. What's missing is evidence of the ability to build operational monitoring infrastructure from scratch.

Um SIEM (Security Information and Event Management) é o núcleo de qualquer operação de segurança. Saber configurar regras de deteção, construir pipelines de ingestão de logs e interpretar alertas é uma competência diretamente aplicável no mercado.

A SIEM (Security Information and Event Management) is the core of any security operation. Knowing how to configure detection rules, build log ingestion pipelines, and interpret alerts is a directly marketable skill.

O objetivo deste projeto foi construir um SIEM completo e funcional com ferramentas gratuitas e open-source, documentando cada decisão técnica — incluindo o que ficaria diferente numa implantação em produção.

The goal of this project was to build a complete, functional SIEM with free and open-source tools, documenting each technical decision — including what would differ in a production deployment.

Decisões de design

Design decisions

Porquê ELK e não Splunk ou Wazuh? O Splunk Free tem um limite de 500MB/dia, o que inviabiliza demos reais. O Wazuh é uma plataforma completa — útil em produção, mas aprende-se menos sobre os componentes individuais. O ELK força a compreensão de cada camada.

Why ELK and not Splunk or Wazuh? Splunk Free has a 500MB/day cap, which kills real demos. Wazuh is a complete platform — useful in production, but you learn less about individual components. ELK forces understanding of each layer.

Porquê Filebeat → Logstash e não Filebeat → Elasticsearch diretamente? Separar transporte de parsing permite atualizar as regras Grok sem tocar no agente de coleta. Em produção, os agentes distribuídos por múltiplos servidores não devem ser atualizados a cada mudança de pipeline.

Why Filebeat → Logstash and not Filebeat → Elasticsearch directly? Separating transport from parsing allows updating Grok rules without touching the collection agent. In production, agents distributed across many servers should not be updated on every pipeline change.

Porquê índices diários? Facilita a gestão de ciclo de vida dos dados — eliminar um dia de dados é apenas apagar um índice. É também como os SIEMs reais funcionam em produção.

Why daily indices? Simplifies data lifecycle management — deleting a day's data is just dropping one index. It's also how real SIEMs work in production.

03 Pipeline de Ingestão Ingestion Pipeline

O coração do projeto é o pipeline Logstash. Os eventos chegam como strings brutas e saem como documentos estruturados com campos normalizados segundo o Elastic Common Schema (ECS).

The heart of the project is the Logstash pipeline. Events arrive as raw strings and leave as structured documents with fields normalized to the Elastic Common Schema (ECS).

Padrões Grok — SSH

Grok patterns — SSH

Logstash DSL logstash/pipeline/siem.conf 
# Extrai campos de linhas como:
# Jun 15 14:23:01 sshd[1234]: Failed password for root from 185.234.219.10 port 22 ssh2

grok {
  match => {
    "message" => "%{SYSLOGTIMESTAMP:event.timestamp} %{HOSTNAME:host.name} sshd\[%{NUMBER:process.pid}\]: Failed password for %{USERNAME:user.name} from %{IP:source.ip} port %{NUMBER:source.port}"
  }
}

# GeoIP: acrescenta país, cidade e coordenadas ao source.ip
geoip { source => "source.ip"; target => "source.geo" }

O Grok é um sistema de expressões regulares com nomes semânticos. %{IP:source.ip} corresponde a qualquer endereço IP válido e guarda-o no campo source.ip. É esta capacidade de nomear campos que transforma texto livre em dados pesquisáveis.

Grok is a regex system with semantic names. %{IP:source.ip} matches any valid IP address and saves it to the source.ip field. This field-naming capability is what turns free text into searchable data.

Enriquecimento GeoIP

GeoIP enrichment

Cada evento com um source.ip é enriquecido com coordenadas geográficas. A base de dados GeoIP vem incluída na imagem Docker do Logstash — sem custo adicional. O resultado é o mapa de origens de ataques no dashboard Kibana.

Each event with a source.ip is enriched with geographic coordinates. The GeoIP database ships inside the Logstash Docker image — no extra cost. The result is the attack-origin map in the Kibana dashboard.

04 Regras de Deteção Detection Rules

O ElastAlert 2 executa queries ao Elasticsearch em intervalos configuráveis e dispara alertas quando as condições das regras são satisfeitas. Foram implementadas três regras que cobrem os três tipos de lógica de deteção mais comuns.

ElastAlert 2 runs queries against Elasticsearch at configurable intervals and fires alerts when rule conditions are met. Three rules were implemented, covering the three most common detection logic types.

Brute Force SSH SSH Brute Force frequency

Dispara quando 10 ou mais tentativas de login falhadas ocorrem do mesmo IP de origem num intervalo de 1 minuto. O threshold de 10 distingue erros humanos (1–3 tentativas) de ferramentas automatizadas como Hydra ou Medusa.

Fires when 10 or more failed login attempts occur from the same source IP within a 1-minute window. The threshold of 10 distinguishes human mistakes (1–3 attempts) from automated tools like Hydra or Medusa.

MITRE ATT&CK: T1110.001 — Brute Force: Password Guessing
Sondagem de Diretórios Web Web Directory Probe any

Dispara em qualquer pedido HTTP a caminhos sensíveis: /admin, /wp-admin, /.env, /.git. Qualquer acesso a estes caminhos num servidor limpo é reconhecimento automático ou um ataque direcionado — não existe falso positivo legítimo.

Fires on any HTTP request to sensitive paths: /admin, /wp-admin, /.env, /.git. Any access to these paths on a clean server is automated recon or a targeted attack — there is no legitimate false positive.

MITRE ATT&CK: T1595.003 — Active Scanning: Wordlist Scanning
Spike de Erros 404 HTTP 404 Spike spike

Dispara quando a taxa de respostas 404 de um IP aumenta 5x em relação à baseline dos últimos 10 minutos. Uma regra de spike adapta-se ao volume de tráfego real — é mais inteligente do que um threshold absoluto que ignora o contexto.

Fires when the 404 response rate from an IP spikes 5x above the 10-minute baseline. A spike rule adapts to real traffic volume — smarter than a flat threshold that ignores context.

MITRE ATT&CK: T1595.003 — Active Scanning: Wordlist Scanning

07 Conclusões Takeaways

→ A separação de responsabilidades entre Filebeat (transporte), Logstash (parsing) e Elasticsearch (armazenamento) não é apenas design elegante — é o que permite escalar e manter cada camada de forma independente em produção. The separation of concerns between Filebeat (transport), Logstash (parsing), and Elasticsearch (storage) is not just elegant design — it's what allows each layer to be scaled and maintained independently in production.
→ Os padrões Grok são o ponto mais frágil do pipeline. Uma linha de log com formato diferente do esperado gera um evento [_grokparsefailure] que não é descartado — é armazenado mas não pesquisável como esperado. Em produção, monitorizar a taxa de falhas de parse é tão importante como monitorizar os alertas. Grok patterns are the most fragile point of the pipeline. A log line with an unexpected format generates a [_grokparsefailure] event that is not discarded — it's stored but not searchable as expected. In production, monitoring the parse failure rate is as important as monitoring alerts.
→ A regra de tipo spike do ElastAlert é significativamente mais robusta do que uma regra de frequency para deteção de scanners. Um threshold absoluto ignora o contexto de tráfego; um múltiplo da baseline adapta-se automaticamente a variações de volume. ElastAlert's spike rule type is significantly more robust than a frequency rule for scanner detection. An absolute threshold ignores traffic context; a baseline multiple adapts automatically to volume variations.
→ O maior constrangimento técnico no WSL2 é o limite de vm.max_map_count. O Elasticsearch falha silenciosamente sem ele. Este tipo de problema específico de ambiente não está na documentação oficial — só se encontra em produção. The biggest technical constraint on WSL2 is the vm.max_map_count limit. Elasticsearch silently fails without it. This type of environment-specific problem is not in official documentation — it's only found in production.
→ Numa iteração futura, integraria Winlogbeat para logs Windows e a API do AbuseIPDB para threat intelligence automática — os dois acréscimos com maior impacto na detecção real com custo ainda nulo. In a future iteration, I would integrate Winlogbeat for Windows logs and the AbuseIPDB API for automatic threat intelligence — the two additions with the highest detection impact at still-zero cost.
